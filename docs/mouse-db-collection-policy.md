# Mouse DB Collection Policy

작성일: 2026-06-13

> 공통 제품 스펙 저장/노출 기준은 [Product Spec Storage & Display Policy](./product-spec-storage-display-policy.md)를 우선 기준으로 삼는다. 이 문서는 마우스 수집 batch에서 그 기준을 적용하는 보조 운영 문서다.

## 1. 목적

마우스 제품 DB 확장은 곧바로 화면에 모든 정보를 노출하는 방식이 아니라, 먼저 넓게 수집하고 검수한 뒤 Finder와 Compare에 필요한 일부만 단계적으로 노출하는 방식으로 운영한다.

- 수집 파일은 실제 제품 DB가 아니다.
- 수집 파일은 `product_config_patch`가 아니다.
- 수집 파일의 값은 제품 TS 데이터 반영 전 수동 검토가 필요하다.
- 공식 출처가 불확실한 항목은 확정 필드처럼 사용하지 않는다.

## 2. 노출 범위 구분

### Finder 일반 필터

Finder 일반 필터는 초보자가 처음 고르는 넓은 기준이다.

- 손 크기
- 마우스 형태
- 무게 선호
- 연결 방식
- 가격 성향

일반 필터는 추천 결과에 직접 영향을 줄 수 있으므로 공식 스펙과 제품 역할이 분명한 경우만 사용한다.

### Finder 상세 필터

Finder 상세 필터는 후보를 조금 더 좁히는 참고 조건이다.

- 연결 상세
- 배터리 또는 충전 방식
- 클릭 소음
- 사이드 버튼
- 휴대성
- 휠 또는 스크롤
- 손목 부담 또는 특수 형태
- 국내 구매 접근성

상세 필터는 강한 탈락 조건으로 쓰지 않고, 확인 가능한 정보가 있을 때만 약하게 참고한다.

### 제품 상세 스펙

제품 상세 스펙은 Compare, 상세 비교, 운영자 검토에 쓰는 정보다.

- 센서명
- 최대 DPI
- IPS
- 가속도
- 폴링레이트
- 스캔율
- 스위치
- 휠 종류
- 코팅
- 크기
- 무게
- 배터리
- 충전 단자
- 수신기 구성
- 소프트웨어
- 온보드 메모리

스펙 수치가 공식 페이지와 판매처마다 다르면 확정값으로 넣지 않고 `rawSpecs.note`와 `uncertainFields`에 남긴다.

### Hidden Field

아래 정보는 초기 화면에 바로 노출하지 않는 hidden 성격의 검토 정보로 둔다.

- 내부 수집 상태
- 이미지 URL 후보
- 이미지 출처 메모
- 판매처별 구성 차이
- 지역별 구성 차이
- 수신기 번들 차이
- 출처 신뢰도
- 추가 확인 필요 필드

`review`, `status`, 이미지 후보, 구매 링크 후보는 검수 전 사용자 화면에 노출하지 않는다.

## 3. 사무용과 게임용 분리 기준

사무용과 게임용은 넓은 필터 하나로 단순 분류하지 않는다. 같은 마우스라도 사용 목적이 겹칠 수 있기 때문이다.

- 넓은 Finder 필터는 형태, 무게, 연결, 가격처럼 비교적 안정적인 기준을 우선한다.
- 게임용, 생산성, 휴대용, 세로형, 조용한 클릭, 스마트 휠, 멀티기기 워크플로우는 `productTags`로 분리한다.
- guide나 Compare 문구에서는 "어떤 목적에 참고할 수 있는지"를 설명하되, 특정 제품을 정답처럼 쓰지 않는다.

## 4. 이미지 수집 정책

이미지는 파일로 저장하지 않는다.

- 이미지 파일 다운로드 금지
- 로컬 이미지 저장 금지
- 확실한 공식 URL이 있을 때만 URL 후보를 수집
- URL 후보가 있더라도 `imageMeta.hidden`은 `true`로 유지
- alt와 sourceNote만 수집해 추후 검토 자료로 사용

이미지 노출은 저작권, 핫링크, 공식 사용 가능 여부를 별도 검토한 뒤 진행한다.

## 5. 불확실한 값 기록 원칙

불확실한 값은 빈 문자열, `null`, 또는 `확인 필요`로 둔다.

- 확정 필드에 추정값을 넣지 않는다.
- `rawSpecs.note`에 왜 불확실한지 적는다.
- `rawSpecs.uncertainFields`에 필드명을 남긴다.
- 판매처별 차이는 `sellerVariantNote`에 남긴다.
- 지역별 차이는 `regionalVariantNote`에 남긴다.
- 수신기와 동글 번들 차이는 `receiverBundleNote`에 남긴다.

특히 무게, 폴링레이트, 수신기 포함 여부, 배터리 구성, 세부 모델명은 판매처와 세대에 따라 달라질 수 있어 보수적으로 기록한다.

## 6. product_config_patch 변환 전 검토

수집 파일은 바로 제품 DB에 넣지 않는다. `product_config_patch`로 변환하기 전에 아래를 수동 확인한다.

1. 기존 DB 중복 여부
2. 공식 제품명과 세대 구분
3. 공식 스펙 출처
4. Finder 일반 필터 값의 안정성
5. Compare에 필요한 요약 가능 여부
6. 이미지와 링크 미노출 유지 여부
7. 금지 표현 포함 여부
8. 불확실한 값이 `rawSpecs.note`에 남아 있는지

수동 검토 후에도 불확실하면 `review` 상태 제품으로 바로 반영하지 않고 수집 상태 또는 보류 후보로 남긴다.
